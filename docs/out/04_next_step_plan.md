# 下一步开发建议

## 推荐顺序

### 1. 收束 `scripts/ui/Main.gd` 的编排边界
- 目标：
  把主控脚本内部至少拆成“节点绑定”“UI 刷新”“回合推进”“局/挑战推进”四组 helper。
- 为什么排第一：
  这是当前所有结构风险的根节点。
- 预期收益：
  后续任何修复都更容易落点，回归范围也更容易控制。

### 2. 清理旧 UI 路径 fallback，并让 smoke 对齐正式场景树
- 目标：
  逐步去掉 `PlayerBetArea`、`BossBetArea`、`BossDeckView`、`PeekBossBetButton` 等旧路径兼容。
- 为什么排第二：
  当前功能已经稳定，正适合从“能兼容跑”过渡到“结构收敛”。
- 预期收益：
  减少 UI 继续背旧包袱，降低后续改场景时的混乱。

### 3. 缩小 `_bind_nodes()` 的 required 集合
- 目标：
  建立主流程必需节点与可选显示节点的分层。
- 为什么排第三：
  这是防止未来 UI 小改动直接把运行链路打断的最小防线。
- 预期收益：
  提升主场景可维护性和多人协作容错率。

### 4. 固化 UI 文本 ownership
- 目标：
  把结果播报、中心引导、Bet 提示、Summary 的刷新规则收束成少数固定入口。
- 为什么排第四：
  当前职责已经初步分开，正适合趁稳定期做“防回退”整理。
- 预期收益：
  降低“同一份信息被写到多个 UI”的风险。

### 5. 把 set / challenge 推进从 UI controller 中抽离
- 目标：
  让 `Main.gd` 更多变成 orchestration + binding 层，而不是唯一状态源。
- 为什么排第五：
  这是价值很高但改动范围也更大的工作，应该放在前四项收束之后。
- 预期收益：
  为后续扩展玩法、存档、回放、调试工具留下空间。

## 每项任务的落地目标

### 任务：主控减负
- 目标：
  不改玩法，只减少 `Main.gd` 的单文件复杂度。
- 为什么现在做：
  主流程已经稳定，适合做结构整理。

### 任务：路径收敛
- 目标：
  用一套正式 UI 树替代“新旧并存”的长期状态。
- 为什么现在做：
  UI 改版已经完成，继续拖只会让兼容层常态化。

### 任务：节点能力分级
- 目标：
  防止一个次要显示节点缺失就中断主流程。
- 为什么现在做：
  这是最小、最稳、收益明确的防回归整理。

### 任务：文本职责收束
- 目标：
  把 popup / 中央提示 / bet hint / summary 的显示边界写死。
- 为什么现在做：
  刚经历一轮显示层稳定，最怕后续把旧问题写回来。

### 任务：状态源下沉
- 目标：
  逐步把 challenge/set/turn 推进从 UI 层剥离。
- 为什么现在做：
  它不是最紧急，但属于决定项目后续可扩展性的关键一项。
